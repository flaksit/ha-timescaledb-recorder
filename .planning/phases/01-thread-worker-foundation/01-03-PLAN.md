**`run()` method** (worker thread entry point):
The run() method implements D-02 and D-03: connect + schema at startup, then enter flush loop. If DB unreachable, log and enter loop anyway so in-memory buffering still works.

```python
def run(self) -> None:
    """Worker thread entry point. Owns the full DB lifecycle."""
    try:
        self._conn = psycopg.connect(self._dsn, autocommit=True)
    except psycopg.OperationalError as err:
        # D-03: DB unreachable at startup — log warning and continue.
        # State events will buffer in the queue; schema + writes attempted on next flush.
        _LOGGER.warning(
            "TimescaleDB unreachable at worker startup: %s — buffering events in memory", err
        )
    else:
        self._setup_schema()

    buffer: list[StateRow] = []
    while True:
        try:
            item = self._queue.get(timeout=5.0)
        except queue.Empty:
            # BUF-03: 5-second flush interval — queue timeout IS the flush timer.
            if buffer and self._conn is not None:
                self._flush(buffer)
                buffer.clear()
            continue

        if item is _STOP:
            # Graceful shutdown: flush any remaining buffered states before exiting.
            if buffer and self._conn is not None:
                self._flush(buffer)
            break
        elif isinstance(item, StateRow):
            buffer.append(item)
        elif isinstance(item, MetaCommand):
            # Flush pending state rows before processing meta — preserves ordering.
            # Example: entity rename must not overtake the last state for that entity_id.
            if buffer and self._conn is not None:
                self._flush(buffer)
                buffer.clear()
            if self._conn is not None:
                self._process_meta_command(item)

    if self._conn is not None:
        self._conn.close()
```

**`_flush()` private method** (WORK-04: Jsonb wrapping):
```python
def _flush(self, buffer: list[StateRow]) -> None:
    """Batch-insert StateRow items using executemany.

    Wraps dict attributes in Jsonb() per WORK-04 — psycopg3 raises TypeError
    if a raw dict is passed to a JSONB column.
    list[str] labels adapt to TEXT[] automatically (no wrapping needed).
    """
    rows = [
        (
            row.entity_id,
            row.state,
            Jsonb(row.attributes),  # WORK-04: required wrapper for JSONB column
            row.last_updated,
            row.last_changed,
        )
        for row in buffer
    ]
    try:
        with self._conn.cursor() as cur:
            cur.executemany(INSERT_SQL, rows)
    except psycopg.OperationalError:
        # Phase 2 adds retry + re-queue; Phase 1 logs and drops to avoid infinite growth.
        _LOGGER.warning(
            "Connection error during flush; %d rows dropped (Phase 2 adds retry)", len(rows)
        )
    except psycopg.DatabaseError as err:
        _LOGGER.error("DB error during flush; %d rows dropped: %s", len(rows), err)
```

**`_process_meta_command()` private method** (SCD2 dispatch):
This method receives a `MetaCommand` and executes the appropriate SCD2 SQL. Change-detection reads use `dict_row` cursor factory so column names match the existing comparison code in syncer's `_*_row_changed` helpers (which use `row["name"]` etc.). The SCD2 snapshot params have a repeated first parameter for the WHERE NOT EXISTS subquery — pass `(*params, params[0])` for snapshot SQL.

```python
def _process_meta_command(self, cmd: MetaCommand) -> None:
    """Execute SCD2 write for a registry change. Called from worker thread."""
    now = datetime.now(timezone.utc)
    registry = cmd.registry
    action = cmd.action

    try:
        if registry == "entity":
            self._process_entity_command(cmd, now)
        elif registry == "device":
            self._process_device_command(cmd, now)
        elif registry == "area":
            self._process_area_command(cmd, now)
        elif registry == "label":
            self._process_label_command(cmd, now)
        else:
            _LOGGER.warning("Unknown registry type in MetaCommand: %s", registry)
    except psycopg.OperationalError:
        _LOGGER.warning("Connection error processing %s %s event", registry, action)
    except psycopg.DatabaseError as err:
        _LOGGER.error("DB error processing %s %s event: %s", registry, action, err)
```

Implement `_process_entity_command`, `_process_device_command`, `_process_area_command`, `_process_label_command` as private methods. Each follows this pattern:

For `_process_entity_command(cmd, now)`:
- `"create"`: execute `SCD2_INSERT_ENTITY_SQL` with `(*cmd.params,)`
- `"remove"`: execute `SCD2_CLOSE_ENTITY_SQL` with `(now, cmd.registry_id)`
- `"update"` with `cmd.old_id is not None` (rename): execute `SCD2_CLOSE_ENTITY_SQL` with `(now, cmd.old_id)`, then `SCD2_INSERT_ENTITY_SQL` with `(*cmd.params,)`
- `"update"` no rename: open `dict_row` cursor, call `self._syncer._entity_row_changed(cur, cmd.registry_id, cmd.params)`, if True: close + insert

For snapshot actions (`"create"` from `async_start`): The `SCD2_SNAPSHOT_ENTITY_SQL` expects the entity_id param twice (once in SELECT, once in WHERE NOT EXISTS). Pass `(*cmd.params, cmd.params[0])` — append `params[0]` (entity_id) as the last positional arg. Add a comment explaining this. Use `SCD2_SNAPSHOT_ENTITY_SQL` instead of `SCD2_INSERT_ENTITY_SQL` for `"create"` action.

Wait — re-reading the design: `async_start()` enqueues `MetaCommand(action="create", ...)` for snapshot and `_handle_entity_registry_updated` also enqueues `action="create"` for live events. The distinction is that snapshots use `SCD2_SNAPSHOT_*_SQL` (idempotent) and live creates use `SCD2_INSERT_*_SQL`. Since both arrive as `action="create"`, we need a way to distinguish. Review D-04 and PATTERNS.md syncer `async_start` pattern: it enqueues with `action="create"` for all snapshot entries.

Resolution: Use `SCD2_SNAPSHOT_*_SQL` for ALL `"create"` actions. This is safe because `SCD2_SNAPSHOT_*_SQL` has `WHERE NOT EXISTS` and is idempotent — a live create for a brand-new entity will also work correctly (no open row exists → insert succeeds). The snapshot SQL is the safer choice for both paths. Document this decision with a comment.

For snapshot SQL, entity params tuple has 13 elements (entity_id at index 0). Pass `(*cmd.params, cmd.params[0])` to supply the 14th `%s` in the WHERE NOT EXISTS subquery. Same pattern for device (9th), area (5th), label (6th).
