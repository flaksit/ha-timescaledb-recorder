<objective>
Extend the `retry_until_success` decorator with three new capabilities while preserving all Phase 2 behaviour: (1) an `on_recovery` hook that fires once after the first success following a stall; (2) an `on_sustained_fail` hook that fires once when cumulative fail duration exceeds a configurable threshold; (3) `on_transient` made optional so the decorator can wrap reads that don't own a connection.

Purpose: This plan unlocks repair-issue auto-clear (OBS-03) and `db_unreachable` detection (OBS-02), both of which Plans 04 and 05 consume. Making `on_transient` optional enables wrapping `_fetch_slice_raw` (which reads through HA's recorder pool — the pool owns the session, not us).

Output: Extended `retry.py` with new signature and closure state; comprehensive tests covering new hook firing rules AND regression of existing behaviour.

Implements: D-03-a (on_recovery hook), D-03-b (on_transient optional), D-03-d (STALL_THRESHOLD from const.py), D-03-e (per-wrapper closure state), D-11 (on_sustained_fail + sustained_fail_seconds).

TDD rationale: This is pure behavioural logic with defined inputs (call results) and outputs (hook invocations). Every new behaviour is expressible as `given a failure pattern, expect hook X fires Y times`. RED→GREEN→REFACTOR is the right cycle here.

Review revision (cross-AI MEDIUM-7): Use `time.monotonic()` for all threshold timing in the retry decorator. Wall-clock time (`time.time()`) can jump backwards (NTP adjustments, DST, VM migration), causing `first_fail_ts` checks to misfire. `time.monotonic()` is guaranteed non-decreasing and is the correct primitive for elapsed-duration comparisons.

Review note (cross-AI HIGH-3 — event loop blocking constraint): The retry decorator uses `stop_event.wait(backoff)` which is a SYNCHRONOUS blocking call. This is correct and intentional for thread worker contexts. However, any call site that uses this decorator from an async context (i.e., where the caller is on the HA event loop) MUST ensure the wrapped function runs inside `hass.async_add_executor_job` or `recorder_instance.async_add_executor_job` — never called directly on the event loop. The decorator docstring must state this constraint explicitly. In this codebase, all current call sites (`_insert_chunk` in states_worker, `_write_item` in meta_worker, `_read_watermark_raw` in states_worker, `_fetch_slice_raw` in backfill via `recorder_instance.async_add_executor_job`) already satisfy this constraint because they run in thread pool contexts. The constraint is a rule for future users of the decorator, not a defect in the current design.

<!-- Existing closure state (retry.py lines 57-102) — the patterns to preserve -->

Existing state per wrapper invocation:
- `attempts: int` (counter reset on success)
- `notified: bool` (stall-notified flag, cleared on success)
- Backoff indexing: `idx = min(attempts - 1, len(backoff_schedule) - 1)`
- Shutdown check: `if stop_event.wait(backoff): return None`

New state to add (D-03-a, D-11):
- `stalled: bool` (arms on_recovery; True when stall threshold crossed, False after on_recovery fires)
- `first_fail_ts: float | None` (monotonic timestamp of first failure in the current failure streak — use `time.monotonic()`, NOT `time.time()`)
- `sustained_notified: bool` (guards on_sustained_fail from firing twice in the same streak)

Event-loop blocking constraint (review HIGH-3):
The `stop_event.wait(backoff)` call is blocking/synchronous. All current call sites in this
codebase run in thread-pool contexts (states_worker thread, meta_worker thread, backfill executor
job) so this is safe. If a future caller wraps a function that is invoked from the event loop
directly, it will block the loop. The decorator docstring must state: "This decorator is
intended for use in dedicated OS threads or executor jobs only. Never call a wrapped
function directly from the HA event loop."
