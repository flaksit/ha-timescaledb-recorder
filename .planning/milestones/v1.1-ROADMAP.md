# Milestone v1.1: Robust Ingestion

**Status:** ✅ SHIPPED 2026-04-23
**Phases:** 1–3
**Total Plans:** 29

## Overview

The v1.1 milestone refactors the write path from an async, event-loop-resident asyncpg design to a dedicated OS worker thread with a synchronous psycopg3 connection. Three phases deliver the full durability story: first the correct thread/async boundary (everything else depends on this), then the durability mechanics (bounded buffer, backfill, metadata recovery, transient error handling), then hardening with watchdog restart and full observability.

**Success Criteria** (all met):
1. Buffer cap holds at 10,000 records; oldest records are dropped (not newest) when the cap is reached; a warning log is emitted with the drop count
2. On startup and on DB-healthy transition, backfill fetches state changes from HA sqlite since `MAX(last_updated)` in `ha_states`, applies the entity filter, and writes in chunks of ≤1000 rows; backfill does not block HA startup
3. Backfill uses per-entity `state_changes_during_period` calls (never `entity_id=None`) dispatched via `recorder_instance.async_add_executor_job`
4. On DB-healthy transition, all four registry tables (entities, devices, areas, labels) are re-snapshotted idempotently via `WHERE NOT EXISTS`; current state is always correct after recovery
5. `PostgresConnectionError`, `OSError`, `DeadlockDetected`, and serialization errors keep rows in the buffer and retry on next flush cycle without logging errors
6. `ON CONFLICT (last_updated, entity_id) DO NOTHING` dedup is only used when TimescaleDB ≥ 2.18.1 is confirmed; a startup check or warning gate enforces this

## Milestone Summary

**Key Decisions:**
- Threaded worker (not async-in-event-loop): matches HA recorder pattern, contains crashes, enables sync psycopg3 — ✓ Good
- psycopg3 over asyncpg: cleaner threading, native JSONB/TEXT[], no async-in-thread complexity — ✓ Good
- Drop-oldest buffer eviction at 10k records: ~6 MB RAM, ~15–20 min coverage; dropped events recoverable from HA sqlite — ✓ Good
- HA sqlite via recorder.history API (not direct SQL): stable public interface, avoids sqlite locking — ✓ Good
- Single queue.Queue replaces threading.Lock: flush and backfill sequential in same worker thread — ✓ Good (simplified Phase 2 significantly)
- Bare psycopg.connect() (not pool): simpler lifecycle; reconnect via watchdog restart path — ✓ Good
- Unified retry path (D-01): dropped SQLERR-03/04 per-row-fallback and critical-log branching; stall notification after N=5 is the recovery story — ✓ Good
- TimescaleDB ≥ 2.18.1 confirmed for ON CONFLICT DO NOTHING dedup (user-confirmed 2026-04-22) — ✓ Good

**Issues Resolved:**
- asyncpg → psycopg3 migration: all JSONB/TEXT[] adapters, placeholder syntax ($N→%s), pool→single-conn lifecycle
- Backfill deadlock risk: state_changes_during_period requires per-entity iteration (entity_id=None raises ValueError)
- Shutdown deadlock: run_coroutine_threadsafe().result() on HA stop → use sentinel + async_add_executor_job(thread.join)
- TimescaleDB ≤ 2.17.2 dedup bug: ON CONFLICT DO NOTHING aborts full batch; gated on version check

**Issues Deferred:**
- SCHEMA-01: schema migration infrastructure (deferred to v2)
- SSL-01: SSL/TLS for TimescaleDB connection (deferred to v2)
- SENSOR-01: sensor entity for integration health (deferred to v2)
- FILTER-01: entity filter in options UI (deferred to v2)

**Technical Debt Incurred:**
- hasattr(ha_recorder, "async_wait_recorder") forward-compat guard: if HA removes this API, recorder_disabled issue won't auto-clear (documented in 03-07 SUMMARY; acceptable, user can dismiss manually)
