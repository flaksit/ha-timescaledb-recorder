<objective>
Extend the backfill path with two Phase 3 capabilities: (1) recorder-retention gap detection at the top of each orchestrator cycle, firing `notify_backfill_gap` when HA's sqlite purged events before TimescaleDB recovered; (2) retry-wrapping of the two read paths (`_fetch_slice_raw` in the recorder pool, `read_watermark` on the worker connection) so transient read errors no longer crash the orchestrator or block the worker loop.

Purpose: Phase 2 left read paths asymmetric from writes — a flaky HA recorder or a brief TimescaleDB hiccup during watermark read would bubble up as an unhandled orchestrator exception (relying entirely on add_done_callback restart). Phase 3 closes that asymmetry so the orchestrator only relaunches on truly unexpected errors. Gap detection surfaces the ONE scenario that retry cannot fix — events that no longer exist in HA's recorder because purge_keep_days elapsed.

Output: `backfill.py` gains gap-detection block and a retry-wrapped `_fetch_slice_raw` at call site (so the closure binds the shared threading stop_event). `states_worker.py` wraps `read_watermark` with the same retry decorator used for `_insert_chunk`. Tests assert gap firing conditions, retry wrapping, and exception-propagation semantics.

Implements: D-08 (gap detection), D-03-c (retry for read_watermark + _fetch_slice_raw), D-09 (OBS-01 folds "backfill failed" into watchdog-recovery path — orchestrator body stays uncaught).

File-ownership note: states_worker.py is also modified by Plan 04 (hooks + outer try/except). Plan 05 runs in Wave 4 AFTER Plan 04 has shipped; the `read_watermark` wrapping is an additive change on top of Plan 04's output.
</objective>
